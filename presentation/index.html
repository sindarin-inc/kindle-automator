Approach & Key Technical Decisions

Platform

Decision Approach Pros Cons Rejected Alternatives
Kindle on emulated Android Run real Kindle app in Android emulator, control via Appium Full catalog, Amazon handles DRM ~600MB RAM per user, latency between action and result
  • Web scraping (obfuscated DOM, unreliable element targeting)
  • Reverse engineering (DRM/legal)
Infrastructure Bare metal x86 emulator (QEMU + KVM), ARM64 Kindle APK translated via libhoudini native bridge KVM acceleration, predictable cost ~30% native bridge translation overhead, no auto-scaling
  • Cloud VMs (nested virtualization too slow or unavailable)
  • ARM64 emulator (Google dropped support after API 25, now on 36)
  • Mac (native ARM emulator works, but no Xvfb — can't isolate per-user VNC displays)

Kindle App Views

Decision Approach Pros Cons Rejected Alternatives
Authentication User enters credentials via VNC No credential storage Requires VNC interaction
  • Store username/password for auto re-login on token loss (security liability)
2FA & CAPTCHA All solved by the user via VNC:
  • Image CAPTCHAs
  • Rotation puzzle
  • Drag puzzle
  • Phone 2FA
  • Email 2FA
 100% reliable Janky VNC'd Android app on a user's iPhone
  • Automated CAPTCHA (brittle, puzzle drag hard)
  • Skip/defer (terrible UX)
Fetching & searching library Scroll library view, scrape metadata from XML, streamed via SSE to be cached by web server Cacheable covers, native book selection in mobile app Minutes for hundreds of books, hour for thousands
Text extraction Mistral OCR primary, Google Doc AI fallback. Firmware switch from relative to absolute paging enabled pre-fetching pages ahead — users never see a loading indicator on the device No Kindle API, most reliable way to get text Non-deterministic, adds latency and cost, munges formatting
  • Tesseract (poor accuracy)

Core Architecture

Decision Approach Pros Cons Rejected Alternatives
View detection XML page source + XPath ~50ms, deterministic, debuggable Brittle; mitigated with fallback chains
  • Vision/ML (slow, needs training data)
  • OCR (5–6s, unreliable)
Navigation Finite state machine, 15 states. Fast expected-view check, then slow detection by likelihood, then app restart as last resort Predictable, new states isolated Upfront modeling required
  • Monolithic if/else (exponential complexity)
Multi-user One AVD per user, keyed by email. New users cloned from a seed AVD for fast init, then device fingerprints randomized (MAC, serial, Android ID) so each clone looks unique True isolation, concurrent, persistent auth Disk + RAM per profile
  • Shared emulator (slow switching)
  • Docker per user (I/O overhead)
Request coordination Redis priority queue + dedup. Some requests cancellable (open-book), some not (navigate mid-tap — can't undo a tap on a physical UI) Higher priority cancels lower, shared results More complex than mutex
  • Naive last-one-wins (non-cancellable requests kept running, caused conflicts)
  • Simple mutex (blocks all concurrent requests)

Tractable Problems

Amazon Detection

Amazon ejects auth tokens from shared device fingerprints. Randomize MAC, serial, Android ID, IMEI per emulator. Fingerprints persist across restarts.

Scale & Resources

~600MB RAM per emulator. Idle shutdown for inactive users, cold storage archival to S3 after 30 days, snapshot recovery for <30s restart, seed clone AVDs for new users.

Zero-Downtime Deploys

Mark running emulators before deploy, pull code, restart server, auto-restore from snapshots. Active users never interrupted.

BLE + WiFi Connectivity

Glasses connect to phone via BLE, phone proxies to server over WiFi. Fallback to direct WiFi when BLE unavailable (hotels, captive portals). Users always reach their Kindle.

Outcome & Impact

Shipped the pivot

Went from "competing with Kindle" to "enhancing Kindle." Unlocked users' existing libraries on hardware Amazon never intended to support.

Sold 1,000 devices

Entire hardware inventory sold in 2–3 weeks after launching Kindle support. Viral TikTok videos showcasing the feature drove organic demand.

964 users on the platform

Real adoption — users authenticated their Amazon accounts, loaded libraries, and read on the glasses daily.

Solo-built, production-hardened

1 engineer, 13 months, 57K lines. Running on 2 bare metal servers serving concurrent users with zero-downtime deploys.

Reflections

  • Betting on the real app paid off Running actual Kindle (not scraping, not reverse engineering) meant we inherited Amazon's catalog, DRM, and sync for free. Every new Kindle feature just works.
  • Partial automation beats flaky full automation VNC for 2FA/CAPTCHA felt like a compromise. It was actually the right call — 100% reliable human auth beats 95% reliable automated auth that fails on a new user's first experience.
  • XPath brittleness is manageable 50ms deterministic detection vs 5–6s non-deterministic OCR. Kindle app updates break XPaths, but fallback chains + forensic XML dumps on failure mean fixes take minutes, not days.
  • The state machine was the best architectural decision When Amazon added puzzle auth, it was one new handler file. Without the FSM, it would have been changes across every flow.

If 3 more months...

  • Vision/ML as fallback when all XPath strategies fail on unknown views
  • Get ARM emulators working — eliminate the 30% libhoudini translation overhead entirely
  • Smarter page pre-caching — OCR page-by-page instead of batch-jumping forward, and sync the placemark back if the client disconnects so Kindle stays where the user was actually reading

If 2 months sooner...

  • Redis request coordination from day one (started single-user, caused significant rework)
  • Skip automated auth/CAPTCHA attempts — go straight to VNC from the start instead of building and then discarding automation for each challenge type
  • Skip snapshot management — just cold boot every time. Slower startup but avoids the snapshot corruption rabbit holes that consumed weeks

Hardest decision

  • Emulating Android over building a bookstore — felt hacky, was right. Users don't care how it works, they care that their library works.
  • VNC for auth instead of automating it — accepting partial automation as a feature, not a limitation.
