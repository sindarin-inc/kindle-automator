The Hardest Parts

Hard Problem 1

Navigating a Moving Target

Amazon updates the Kindle app constantly — new dialogs, changed layouts, different auth flows. Every update is a mini-crisis. Solution: the state machine was designed for change. Each view has multiple fallback XPath strategies. 50+ dialogs intercepted. Unknown states dump XML + screenshot for forensic diagnosis, and fixes take minutes because the view detection layer is isolated from the action logic.

Hard Problem 2

Concurrent Request Coordination

Glasses poll screenshots every few seconds while the web app opens a book or navigates. One emulator, many callers. Solution: Redis priority queue with cancellation propagation. shutdown=200 > open-book=100 > navigate=50 > screenshot=0. Higher priority cancels lower mid-execution. Identical requests deduplicate and share results. "Last-one-wins" for book opens.

Hard Problem 3

Human-in-the-Loop Authentication

Not just initial login — the full auth lifecycle. Amazon rotates 5 different challenge types (image CAPTCHA, rotation puzzle, drag puzzle, phone 2FA, email 2FA), revokes tokens unpredictably, and changes auth flows between app versions. Solution: VNC streaming infrastructure (WebSocket proxy, rfbproxy) lets users solve challenges on their phone. The state machine waits in an auth state, detects completion, and resumes the interrupted operation.

Hard Problem 4

Making Emulators Work Headlessly at All

365 of 1,424 commits are emulator-related. The Kindle app is ARM64; Google dropped ARM emulator support after API 25 (we need 36). So: x86 emulators with QEMU binary translation of the ARM APK. Headless on bare metal means no GPU — software rendering, Xvfb virtual displays, VNC servers per instance. Appium crashes, ADB disconnects, emulators hang on boot, snapshots corrupt between graphics pipeline versions. Each of these was a multi-day investigation. There's no documentation for running Kindle in a headless x86 emulator with ARM translation — this configuration basically doesn't exist outside this project.

Tractable Problems

Amazon Detection

Amazon ejects auth tokens from shared device fingerprints. Randomize MAC, serial, Android ID, IMEI per emulator. Fingerprints persist across restarts.

Scale & Resources

~600MB RAM per emulator. Idle shutdown for inactive users, cold storage archival to S3 after 30 days, snapshot recovery for <30s restart, seed clone AVDs for new users.

Zero-Downtime Deploys

Mark running emulators before deploy, pull code, restart server, auto-restore from snapshots. Active users never interrupted.

Reflections

  • Betting on the real app paid off Running actual Kindle (not scraping, not reverse engineering) meant we inherited Amazon's catalog, DRM, and sync for free. Every new Kindle feature just works.
  • Partial automation beats flaky full automation VNC for 2FA/CAPTCHA felt like a compromise. It was actually the right call — 100% reliable human auth beats 95% reliable automated auth that fails on a new user's first experience.
  • XPath brittleness is manageable 50ms deterministic detection vs 5–6s non-deterministic OCR. Kindle app updates break XPaths, but fallback chains + forensic XML dumps on failure mean fixes take minutes, not days.
  • The state machine was the best architectural decision When Amazon added puzzle auth, it was one new handler file. Without the FSM, it would have been changes across every flow.

If 3 more months...

  • Vision/ML as fallback when all XPath strategies fail on unknown views
  • Abstract away the emulator behind an API proxy — clients shouldn't know it's UI automation
  • Automated library indexing instead of scroll-scraping (minutes for hundreds of books is too slow)

If 2 months sooner...

  • Redis request coordination from day one (started single-user, caused significant rework)
  • List-only library view (skip grid view support — simpler scraping)
  • Fewer initial states — start with LIBRARY, READING, AUTH, add others as encountered

Hardest decision

  • Emulating Android over building a bookstore — felt hacky, was right. Users don't care how it works, they care that their library works.
  • VNC for auth instead of automating it — accepting partial automation as a feature, not a limitation.
